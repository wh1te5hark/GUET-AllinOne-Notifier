# 贡献注意事项

`notifier_dev_doc` 目录是开发文档

## 基本原则

- 变更最小化：优先做最小可验证修改，避免一次性大面积重写。
- 前后端边界清晰：前端分支不应修改后端公共 API 语义。

## API 语义一致性

- 不要在前端优化提交中修改后端路由入口约定（例如把 `GET /` 改为其他语义）。
- 若确需调整 API 约定，必须：
  - 单独提交并在 PR 标题中明确说明；
  - 提供兼容策略与迁移说明；
  - 在文档中标明影响范围和回滚方案。

## 前端规范

- 尽量不要用紫色调配色
- 保持界面简洁清晰，避免过度动画与复杂视觉特效。
- 需兼顾响应式布局和主流浏览器兼容性。
- 若引用外部静态资源，优先本地化

## 后端和采集器开发要求

- 涉及接口调用的开发必须实测，不允许凭猜测实现。
- 新增接口/数据库结构时，同步更新文档并补充必要注释。

## 提交流程建议

1. 先运行基本检查（如 `ruff`、`pytest` 或前端关键流程自测）。
2. 自查是否引入越界修改（前端改后端、后端改前端部署行为）。
3. 更新文档（README 或相关文档/说明）后再提交。
